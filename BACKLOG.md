## クロスフォルダ pHash 画像検索

**問題**: モデルが SNS 投稿した写真がローカルのどの撮影会フォルダのどのバンドル
かを探すのが大変。SNS 側でファイル名は失われ、Instagram などは EXIF も剥がす。

**狙い**: SNS 画像（スクショ等）をアプリに渡したら、ローカルの撮影会フォルダ群
横断で「これ」というバンドルを top-N でランクして提示。クリックで投稿レコードを
そのバンドルに紐付け。

### 設計

- 検索インデックスは **各撮影会フォルダの `.photoorg/index.json` に pHash を
  埋め込む**形で分散保持。アプリは「ライブラリ」概念を持たない。
- ユーザーは検索時に「検索ルート」（例: `D:/Photography/`）を指定 → アプリが
  サブフォルダ再帰で `.photoorg/index.json` を集めて pHash 群を集約 → 対象画像
  との **ハミング距離** でランキング。
- 実写真ファイルにはアクセスしないので、数千バンドル横断でも数百ms 想定。

### 実装フェーズ

| フェーズ | やること | 依存 |
|---|---|---|
| 1 | scan 時に pHash 計算 → `.photoorg/index.json` に保存 | — |
| 2 | 横断検索 Tauri コマンド: ルート → 全 index.json 集約 → 距離計算 → トップN | 1 |
| 3 | 検索ダイアログ UI: ルート指定 + 画像ドロップ + 結果リスト + 「投稿追加」ボタン | 2 |
| 4 | クロスフォルダ投稿追加: `save_bundle_sidecar` を別フォルダ宛に発行 | 3 |

### 注意点

- pHash 未計算の旧 scan フォルダは検索対象外 → 結果画面に「⚠ N フォルダが
  pHash 未計算（再スキャンで対象化）」を出す
- 検索ルートは設定で記憶
- DCT ベース pHash は強い JPEG 再圧縮・リサイズに耐性あり、中程度のクロップは弱い
- ガチ精度版（CLIP埋め込み等）はオーバーキル。pHash で 80% カバーできれば十分

### 関連したいユースケース

- 検索結果から「このバンドルがあるフォルダを開く」もできるとよい（モデル投稿
  確認後にそのフォルダで他の作業をしたいケース）
